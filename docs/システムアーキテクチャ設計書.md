# システムアーキテクチャ設計書

紅白予想アプリ（kohaku-yoso）の全体構成・技術選定・実行環境を記述する。

## 1. 設計方針

- **全てを Cloudflare 上で完結させる**。コストを最小化し、無料枠で運用開始できることを優先する。
- MVPはシンプルに保ち、過度な作り込みを避ける。完璧なアーキテクチャより動くMVPを優先する。
- 外部サービスは原則増やさない。ブロックチェーンは譲渡不可の記念トークン（ソウルバウンド）のみ例外として採用する。
- リアルマネーの賭け・換金性は持たせない。

## 2. 技術スタック

> 画面数が少ないため、ルータ無しのタブ切替・プレーンCSS・TanStack Query 中心の構成になっている。
> UIローカル状態は `useState` で足りており Zustand は未導入。

## 3. 全体構成図

```
                ┌──────────────────────────────────────────────┐
   ブラウザ ───▶│  Cloudflare Worker（同一オリジン）             │
  (React SPA)   │                                              │
                │  /api/*  /health  → Hono アプリ (server)      │
                │  それ以外          → 静的アセット (web/dist)   │
                │                      SPAフォールバック         │
                └───────────────┬──────────────────────────────┘
                                │
            ┌───────────────────┼─────────────────────┐
            ▼                   ▼                     ▼
      Cloudflare D1        外部 OAuth/RPC         (任意) KV
       (SQLite)        Google OAuth / Spotify   セッション保存
                       MusicBrainz / Solana RPC  （現状は未使用）
```

本番では **API と SPA を同一オリジンで配信**する（`wrangler.jsonc` の `env.production.assets`）。
同一オリジンにすることで Cookie / OAuth リダイレクトが単純化される。
`/api/*` と `/health` は Worker を先に実行（`run_worker_first`）し、それ以外は SPA にフォールバックする。

```txt

   ├─ drizzle/            # マイグレーション(SQL)・スナップショット
   ├─ seed.sql            # サンプルデータ
   └─ src/
      ├─ index.ts         # Workersエントリ（Honoアプリ・ルーティング集約）
      ├─ routes/          # HTTPハンドラ（auth/predictions/artists/songs/admin/rankings/wallet/seasons）
      ├─ services/        # 横断ロジック（externalMusic, search, distribution）
      ├─ domain/          # 純粋ロジック（scoring, ranking, season, search, wallet, distribution）
      ├─ repositories/    # D1アクセス（Drizzle）
      ├─ schemas/         # Zodスキーマ（admin, predictions, wallet）
      ├─ auth/            # google, session, admin, wallet-challenge
      ├─ token/           # Solana minter（factory / minter / solana-minter）

      ├─ lib/             # http（エラー整形）
      └─ types/           # Env（バインディング）など
```

### レイヤリング

`routes → services / domain / repositories` の単方向依存を基本とする。

- **routes**: HTTP入出力・認可ミドルウェア・Zod検証・エラー整形。ビジネス判断は持たせない。
- **domain**: DB非依存の純粋関数（採点・ランキング・受付判定・検索正規化・署名検証）。テストが厚い層。
- **services**: 複数リポジトリや外部APIをまたぐ処理（外部音楽検索、トークン配布のオーケストレーション）。
- **repositories**: D1（Drizzle）への CRUD。
- **schemas**: Zodスキーマ。`z.infer` で型を導出し型とバリデーションを二重管理しない。

## 5. 認証・セッション

- **方式**: Google OAuth 2.0（OpenID Connect）。自前パスワードは持たない。
- **ユーザー識別**: Google の `sub`（`users.google_sub`）を正とする。メールは変更され得るため照合キーにしない。
- **ログイン導線**:
  1. `GET /api/auth/google/login` … `state` を生成し HttpOnly Cookie に保存、Google認可画面へリダイレクト。
  2. `GET /api/auth/google/callback` … `state` 照合 → 認可コードをトークン/プロフィールに交換 →
     `google_sub` で `users` を find-or-create → セッション発行 → `POST_LOGIN_REDIRECT`（既定 `/`）へ。
- **セッション**: ユーザーIDを JWT(HS256, `SESSION_SECRET` 署名) にして `session` Cookie へ格納。
  - 属性: `HttpOnly` / `SameSite=Lax` / `Secure`（httpsのみ） / `Path=/` / 有効期限 30日。
  - 検証ミドルウェア `requireAuth`（未ログインは 401）。書き込み系API・外部検索APIに適用。
- **管理者**: `users.is_admin` フラグ。`requireAdmin` ミドルウェアで `/api/admin/*` 全体を保護（非管理者は 403）。
  管理者化はDB直接更新で行う（管理画面は未提供）。
- **KV**: セッションをKVに保存する選択肢は残してあるが（`Bindings.SESSIONS?`）、現状は JWT Cookie のみで KV 未使用。

## 6. 外部連携

| 連携先          | 用途                        | 必須/任意 | 備考                                        |
| --------------- | --------------------------- | --------- | ------------------------------------------- |
| Google OAuth    | ログイン                    | 必須      | `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET` |
| Spotify Web API | 予想時のアーティスト/曲検索 | 任意      | 未設定なら MusicBrainz にフォールバック     |
| MusicBrainz     | 同上のフォールバック        | 任意      | User-Agent 必須。鍵不要                     |
| Solana RPC      | 記念トークンの mint/送付    | 任意      | 未設定なら配布機能は 501 で無効             |

外部音楽検索は Spotify を主、MusicBrainz をフォールバックとする（`services/externalMusic.ts`）。
取り込むのは自DBに必要な最小限（名前・URL・リリース年など）のみ。

## 7. 環境変数・シークレット

ローカルは `server/.dev.vars`、本番は `wrangler secret put` で設定する（いずれもコミットしない）。
`Bindings` 型（`server/src/types/env.ts`）が一覧の正。

| 変数                                          | 必須 | 用途                                         |
| --------------------------------------------- | ---- | -------------------------------------------- |
| `DB`（D1バインディング）                      | 必須 | アプリDB（`wrangler.jsonc`）                 |
| `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET`   | 必須 | Google OAuth                                 |
| `SESSION_SECRET`                              | 必須 | JWTセッション署名・ウォレットチャレンジ署名  |
| `OAUTH_REDIRECT_URI`                          | 任意 | 未設定ならリクエストのオリジンから導出       |
| `POST_LOGIN_REDIRECT`                         | 任意 | ログイン後の遷移先（既定 `/`）               |
| `SPOTIFY_CLIENT_ID` / `SPOTIFY_CLIENT_SECRET` | 任意 | Spotify検索。未設定で MusicBrainz のみ       |
| `SOLANA_RPC_URL`                              | 任意 | 記念トークン用 RPC（devnet）                 |
| `TOKEN_MINT_ADDRESS`                          | 任意 | 記念トークンの mint アドレス                 |
| `MINT_AUTHORITY_SECRET`                       | 任意 | mint権限の秘密鍵（base58。最重要・取扱注意） |

## 8. デプロイ・実行

- ローカル開発: `pnpm dev:server`（`wrangler dev`, :8787） / `pnpm dev:web`（Vite, :5173）。
- DB: `pnpm --filter @kohaku/server db:generate`（マイグレーション生成） /
  `db:migrate:local|remote` / `db:seed:local`。
- 本番: `web` をビルドして `web/dist` を生成 → `wrangler deploy --env production`
  （Worker が API と SPA を同一オリジンで配信）。詳細は [`/DEPLOY.md`](../DEPLOY.md)。

## 9. テスト戦略

- ランナーは Vitest。`pnpm test`（= `@kohaku/server` の `vitest run`）。
- **純粋ドメインロジック**（採点・早押し係数・ランキング・受付判定・検索正規化・署名検証）は
  DB非依存で単体テストする（`domain/*.test.ts`）。採点は AGENTS.md のスコア表の全ケース＋境界を網羅する。
- ルート/結合は `routes/*.test.ts`、サービスは `services/*.test.ts` でカバーする。
- 手動の総合試験は [総合試験仕様書](./総合試験仕様書.md)。
