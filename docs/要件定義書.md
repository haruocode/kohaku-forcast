# 要件定義書

紅白予想アプリ「紅白予想（kohaku-yoso）」の要件を定義する。
プロダクト方針の正は [`/AGENTS.md`](../AGENTS.md)。本書はそれを要件として整理したもの。

## 1. 背景・目的

NHK紅白歌合戦の「誰が出場するか」「何を歌うか」「結果はどうなるか」を予想して楽しむWebアプリのMVP。
リアルマネーのギャンブル要素は持たせず、紅白向けの軽量な予想マーケット／ファンタジーリーグのような
体験を提供する。コストを最小化し、Cloudflareの無料枠で運用開始できることを優先する。

## 2. スコープ

### 2.1 対象（MVP）

- アーティスト・曲の検索と予想の投稿・閲覧
- 確信度・任意コメント付きの予想
- 結果確定後のスコアリングとランキング（シーズン／通算）
- サイト内ポイント（採点スコア。シーズン／通算）
- 年度（シーズン）単位の運用
- Google認証

### 2.2 対象外（割り切り）

- リアルマネーの賭け・換金性のある仕組み（方針として恒久的に作らない）
- ブロックチェーン／トークン（記念トークン含む）。実績はサイト内ポイントで表す
- 換金・購入・景品交換できるポイント。予想にコスト（賭け金）を課す仕組み
- 高度なあいまい検索（類似度・タイプミス自動補正）。部分一致＋かな正規化で割り切る
- ネイティブアプリ、プッシュ通知、SNS連携、オッズ可視化（将来構想）

## 3. ユーザー区分

| 区分               | 説明                    | 権限                                                              |
| ------------------ | ----------------------- | ----------------------------------------------------------------- |
| 未ログインユーザー | 閲覧のみ                | 一覧・検索・ランキングの閲覧                                      |
| 一般ユーザー       | Googleログイン済み      | 予想の投稿・編集・取消、ランキング閲覧                            |
| 管理者             | `is_admin=1` のユーザー | 上記＋シーズン/アーティスト/曲/結果の管理、締切操作              |

## 4. 機能要件（概要）

詳細・一覧は [機能一覧](./機能一覧.md) を参照。

- **FR-1 認証**: Google OAuth 2.0（OIDC）でログイン/ログアウト。`google_sub` で同一人物を識別。
- **FR-2 検索**: アーティスト/曲をローカルDB（部分一致・かな正規化）と外部音楽DB（Spotify/MusicBrainz）で検索。
- **FR-3 予想**: 受付中シーズンに対し、アーティスト（必須）・曲（任意）・確信度・コメントで予想を投稿。
  同一シーズン・同一アーティストの二重投稿は不可。本人のみ受付中に編集・取消できる。
- **FR-4 受付管理**: 締切（公式発表日時）は運営が後から設定する。締切後の投稿・編集・取消は拒否する。
- **FR-5 結果確定**: 管理者が各アーティストの出場可否・歌唱曲を確定する（冪等）。
- **FR-6 スコアリング**: 予想×結果から加点（早押しボーナス込み）・減点を算出する。
- **FR-7 ランキング**: 締切後にシーズンのランキングを表示する。締切済み全シーズンを合算した通算ランキングも表示する。
- **FR-8 サイト内ポイント**: 採点スコアをポイントとして扱う。シーズンポイントと通算ポイント（全締切済みシーズン合計）を持つ。譲渡・換金・購入いずれも不可。

## 5. 業務ルール（重要要件）

### 5.1 予想の受付ルール

- `prediction_close_at` が NULL（受付中）または `now < close_at` のときのみ投稿・編集・取消が可能。
- 締切は運営が「公式発表の日時」として後から設定する。**操作時点より過去の日時も設定できる**（現実の発表時刻が正）。
- **採点対象は `predictions.updated_at < close_at` の予想のみ**。公式発表時刻以降に投稿・編集された
  予想は不正投票として採点・ランキングから除外する（物理削除せず採点時に除外）。
- すべての可否チェックはサーバー側で行う（フロントの表示制御に依存しない）。

### 5.2 スコアリング

加算方式。アーティストと曲を独立評価して合算する。

| 予想内容         | 結果        | 基礎点 |
| ---------------- | ----------- | ------ |
| アーティスト＋曲 | 出場○ / 曲○ | +30    |
| アーティスト＋曲 | 出場○ / 曲✗ | +10    |
| アーティスト＋曲 | 出場✗       | -10    |
| 出場予想のみ     | 出場○       | +10    |
| 出場予想のみ     | 出場✗       | -10    |

- 外れ -10 は的中 +10 と対称（損益分岐＝出場率50%）にし、撃ち得を防ぐ。合計はマイナスを許容する。
- **早押しボーナス**: 最終点 = 基礎点 ×（1 + 0.5 × 早さ係数）。受付直後の的中は最大1.5倍、締切間際は1.0倍。
  ボーナスは正の基礎点のみに掛かる（外れ -10 はフラット）。早さは `updated_at` で測る。
- 確信度（1〜5）はMVPでは採点に影響しない（表示用）。

### 5.3 ランキング

並び順: ①合計スコア降順 → ②的中件数降順 → ③最初の予想投稿が早い順。スコア・的中件数が同じなら同順位。
通算ランキングは締切済み全シーズンのスコア・的中件数を合算し、同じ並び順で順位付けする。

### 5.4 サイト内ポイント

- 採点スコアをそのままポイントとする。**シーズンポイント**（そのシーズンの合計）と
  **通算ポイント**（締切済み全シーズンの合計。マイナスも許容）を持つ。
- **譲渡・換金・購入いずれも不可**。財産的価値を持たせず、賭けにも使わせない。
- 専用テーブルは持たず、予想×結果から都度算出する（紅白は年1回でデータ件数が小さい）。

## 6. 非機能要件

| 区分         | 要件                                                                                                |
| ------------ | --------------------------------------------------------------------------------------------------- |
| コスト       | Cloudflareの無料枠で運用開始できること。外部サービスを増やさない                                    |
| 可用性       | Cloudflare Workers / D1 / Pages の標準SLAに準ずる                                                   |
| 性能         | 紅白規模（データ件数は小）を前提。検索は体感で許容できる速さ                                        |
| セキュリティ | セッションは HttpOnly Cookie（JWT/HS256）。書き込み系・外部検索は認証必須。秘密情報はコミットしない |
| 監査性       | 不正投票は物理削除せず、採点時に除外して記録を残す                                                  |
| 保守性       | TypeScript strict、ドメインロジックとルートの分離、ロジックにテスト                                 |
| 国際化       | UI・エラーメッセージは日本語                                                                        |
| 対応環境     | モダンブラウザ。スマホ幅でレイアウトが崩れないこと                                                  |

## 7. 制約・前提

- データベースは Cloudflare D1（SQLite）。`pg_trgm` 相当の類似検索は使えない。
- 認証は Google OAuth のみ（自前パスワードは持たない）。
- 実績・順位はサイト内ポイントで表す。ブロックチェーン／トークンは使わない。
